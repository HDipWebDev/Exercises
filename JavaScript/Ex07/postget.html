We have been dealing with JavaScript, so there was very little that we had to do in order to access the information inputted by the user.

The reality for many (if not most) web applications is that the information the user inputs will be used by a controlling script of some kind, typically php. The controlling script is likely to enter that information into a database, or use the information in some other way

The question then becomes 'How do I send data to the server'. In essence the server takes requests, executes them, then serves up the data. So what we do is simply 'attach' our form data to the request. We have two ways to do this. The 'post' method and the 'get' method.

POST Method (little bit secure)

POST under HTTP is by no means secure. It simply attaches your form data to the HTTP request headers. This means that your data is transmitted over HTTP in plain text, and is open to anyone who cares to have a look. HTTPS is widly accepted as a secure. It should be noted however that the entire site must be handled through HTTPS, not just the sensitive areas.

Production sites generally use the POST method as it is a little more secure. GET is useful for debugging.

GET Method (not secure)

The GET method puts the form data directly onto the URL of the request. This approach allows for bookmarking and other functionality, however it lacks security. It should not be used for sensitive data, such as username, password, bank details. etc.

Why do we care?

Using GET, of itself is not such a big deal. The problem in this instance is that the web authors appear to be using an MVC model for the site. OK, that is good. But when you use the search bar it puts the search data onto the URL for all to see. This means that somewhere something did not get controlled. If you mess with the site a bit, it will also throw a 400 page that tells everyone exactly what the site is running on.
